## 1. プロジェクト概要

### 1.1 目的
就職活動における企業ごとの選考ステータスや各種締切を一元管理・可視化するWebアプリケーション。
Googleアカウントによるシングルサインオンで、PC・スマートフォンを問わず即座にアクセスでき、リアルタイムにスケジュールを確認・更新できることを主眼とする。

### 1.2 対象ユーザー
- 就職活動中の学生・求職者

### 1.3 利用環境
- Webブラウザ（Chrome / Firefox / Safari / Edge 最新版）
- スマートフォンブラウザ（レスポンシブ対応）

### 1.4 本番環境
| レイヤー | サービス | URL |
|---------|---------|-----|
| フロントエンド | Vercel | https://job-track-neon.vercel.app |
| バックエンド | Render | https://jobtrack-api-rs0s.onrender.com |
| データベース | Neon（PostgreSQL） | ap-southeast-1（シンガポール） |

## 3. 非機能要件

### 3.1 パフォーマンス
| 項目 | 目標値 |
|------|--------|
| 画面初回表示 | 2秒以内（LCP） |
| APIレスポンス | 200ms以内（P95） |
| カンバンドラッグ操作 | 楽観的更新で即時反映（ネットワーク待ちなし） |
| 直近予定自動更新 | デフォルト10分間隔（設定変更可） |

### 3.2 セキュリティ
| 項目 | 対策 |
|------|------|
| 認証トークン検証 | FastAPI側でAuth.js発行JWTの署名・有効期限を検証 |
| CORS設定 | 許可オリジンをフロントエンドのドメインのみに限定 |
| SQLインジェクション対策 | SQLAlchemy ORMのパラメータバインドを使用 |
| XSS対策 | Next.jsのデフォルトエスケープ |
| HTTPS強制 | 本番環境ではVercel・RenderともにHTTPS必須 |
| ユーザーデータ分離 | 全APIエンドポイントでJWTのuser_idとDBのuser_idを照合 |

### 3.3 可用性・運用
| 項目 | 方針 |
|------|------|
| 開発環境 | ローカル（Python venv + npm run dev） |
| 本番環境 | Vercel（フロントエンド）+ Render（バックエンド）+ Neon（DB） |
| DBマイグレーション | Alembic を使用。マイグレーションファイルをGit管理 |
| CI/CD | GitHubへのプッシュでVercel・Renderが自動デプロイ |
| スリープ対策 | Renderの無料プランはUptimeRobotで定期死活監視推奨 |

### 3.4 レスポンシブデザイン
- モバイルファースト設計（Tailwind CSS のブレークポイントを活用）
- カンバンボードはモバイルでは横スクロール

## 7. 制約・前提条件

- ユーザーはGoogleアカウントを保有していること
- データは個人利用を前提とし、チーム共有機能は対象外
- Renderの無料プランは15分無操作でスリープ（初回アクセスに30〜60秒かかる）
- メール取り込みにはAnthropic APIキーが必要
